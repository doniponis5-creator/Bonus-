## Что НУЖНО ДОБАВИТЬ (эти поля сейчас НЕ отправляются)

### 1. `total_amount` — Общая сумма рассрочки
Полная сумма документа реализации. Сколько клиент должен был заплатить изначально.
```
total_amount = СуммаДокумента (из Реализации товаров и услуг)
```

### 2. `paid_amount` — Сколько уже оплатил
Сумма всех оплат по этому документу.
```
paid_amount = total_amount - amount
```
Или можно посчитать напрямую из документов оплаты (ПКО, банковские выписки) привязанных к этой реализации.

### 3. `overdue_days` — Дни просрочки
Сколько дней клиент просрочил оплату. Если не просрочил — 0.
```
Если есть просроченный платёж по графику:
  overdue_days = (ТекущаяДата() - ДатаПросроченногоПлатежа) в днях
Иначе:
  overdue_days = 0
```

### 4. `schedule` — График платежей
Массив запланированных платежей. Берётся из графика рассрочки.

Каждый элемент:
```json
{
  "date": "2026-06-10",      // Дата платежа (формат yyyy-MM-dd)
  "amount": 13500,            // Сумма платежа
  "status": "pending"         // "pending" = будущий, "overdue" = просроченный
}
```

Логика определения status:
- Если дата платежа < Сегодня И платёж НЕ оплачен → `"overdue"`
- Иначе → `"pending"`

**Где взять график:**
- Если используется типовой механизм рассрочки в УТ 11.5 — из регистра "ГрафикОплатыЗаказов" или из условий оплаты документа
- Если рассрочка ведётся через этапы оплаты — оттуда
- Если вручную задаётся — нужно договориться с бухгалтером где хранится

### 5. `payments_history` — История фактических оплат
Массив реальных оплат которые клиент уже сделал.

Каждый элемент:
```json
{
  "date": "2026-05-10",
  "amount": 13500,
  "document": "Приходный кассовый ордер 00ЦБ-001230 от 10.05.2026",
  "overdue_days": 5
}
```

`overdue_days` — на сколько дней просрочил оплату (0 = оплатил вовремя).
Считается так: `ДатаФактическойОплаты - ДатаПлановогоПлатежа` в днях.
Если оплатил раньше или вовремя — `0`.

**Где взять:**
- Приходные кассовые ордера (ПКО) привязанные к этой реализации
- Оплаты от покупателя (банк)
- Любые документы оплаты по взаиморасчётам с этим контрагентом по этому договору/заказу

### 6. `next_payment` — Ближайший следующий платёж
Один объект — ближайший будущий платёж из графика.
```json
{
  "date": "2026-06-10",
  "amount": 13500
}
```
Берётся из `schedule` — первый элемент где `status = "pending"` и `date >= Сегодня`.
Если все платежи просрочены — берём ближайший просроченный.
Если рассрочка погашена — `null`.

## ВАЖНЫЕ ПРАВИЛА

1. **`reference`** — НЕ МЕНЯТЬ формат! Это уникальный ключ. Должен совпадать с тем что уже отправляется.

2. **`phone`** — формат `+996XXXXXXXXX`. НЕ МЕНЯТЬ.

3. **Каждая рассрочка = отдельный POST запрос.** Если у клиента 3 рассрочки — 3 запроса. Это уже так работает, не менять.

4. **Если у рассрочки НЕТ графика** (обычный долг без рассрочки) — отправляй `schedule: []`, `payments_history: []`, `next_payment: null`. Это нормально.

5. **`paid_amount`** можно считать как `total_amount - amount` если нет точных данных об оплатах.

6. **`overdue_days`** — если не можешь точно посчитать, отправляй `0`. Лучше 0 чем неправильное число.

7. **Формат дат** — строго `yyyy-MM-dd` (например `2026-06-10`). НЕ `dd.MM.yyyy`!

## МИНИМАЛЬНАЯ ДОРАБОТКА (если нет графика рассрочки в 1С)

Если в 1С нет типового графика рассрочки, можно отправить хотя бы:

```json
{
  "phone": "+996555353042",
  "full_name": "ФИО",
  "amount": 67500,
  "total_amount": 81000,
  "paid_amount": 13500,
  "overdue_days": 0,
  "reference": "Реализация товаров и услуг 00ЦБ-000621 от 05.04.2026",
  "note": "Рассрочка | ФИО",
  "schedule": [],
  "payments_history": [],
  "next_payment": null
}
```

Это уже даст клиенту видеть: общую сумму, сколько оплатил, сколько осталось, процент оплаты.

## ГДЕ ИСКАТЬ ДАННЫЕ В 1С:УТ 11.5

| Поле | Где в 1С |
|------|----------|
| `total_amount` | `Реализация.СуммаДокумента` |
| `paid_amount` | Сумма связанных ПКО/оплат, или `total_amount - amount` |
| `overdue_days` | Разница дней между сегодня и просроченной датой из графика |
| `schedule` | Регистр `ГрафикОплатыЗаказов`, или этапы оплаты, или условия рассрочки |
| `payments_history` | ПКО + Оплаты от покупателей привязанные к реализации |
| `next_payment` | Первый будущий платёж из schedule |
